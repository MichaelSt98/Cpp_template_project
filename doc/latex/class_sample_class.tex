\href{https://www.learncpp.com/cpp-tutorial/9-14-overloading-the-assignment-operator/}{\texttt{ Overloading assignment operator}}\hypertarget{class_sample_class_autotoc_md164}{}\doxysubsubsection{Shallow vs. deep copy}\label{class_sample_class_autotoc_md164}
The default copy mechanism for classes is {\bfseries{memberwise}} copy (also called {\bfseries{shallow copy}}), which works for simple classes, without dynamically reserved memory, very good.

However, a {\bfseries{deep copy}} allocates memory for the copy and then copies the actual value, so that the copy lives in distinct memory from the source. This requires to write copy constructors and overloaded assignment operators.


\begin{DoxyItemize}
\item The default copy constructor and default assignment operators do shallow copies, which is fine for classes that contain no dynamically allocated variables.
\item Classes with dynamically allocated variables need to have a copy constructor and assignment operator that do a deep copy.
\item Favor using classes in the standard library over doing your own memory management.
\end{DoxyItemize}\hypertarget{class_sample_class_autotoc_md165}{}\doxysubsubsection{Object relations}\label{class_sample_class_autotoc_md165}

\hypertarget{class_sample_class_autotoc_md166}{}\doxyparagraph{Composition}\label{class_sample_class_autotoc_md166}
To qualify as a composition, an object and a part must have the following relationship\+:


\begin{DoxyItemize}
\item The part (member) is part of the object (class)
\item The part (member) can only belong to one object (class) at a time
\item The part (member) has its existence managed by the object (class)
\item The part (member) does not know about the existence of the object (class)
\end{DoxyItemize}

Therefore\+:


\begin{DoxyItemize}
\item Typically use normal member variables
\item Can use pointer members if the class handles object allocation/deallocation itself
\item Responsible for creation/destruction of parts
\end{DoxyItemize}\hypertarget{class_sample_class_autotoc_md167}{}\doxyparagraph{Aggregation}\label{class_sample_class_autotoc_md167}
To qualify as an aggregation, a whole object and its parts must have the following relationship\+:


\begin{DoxyItemize}
\item The part (member) is part of the object (class)
\item The part (member) can belong to more than one object (class) at a time
\item The part (member) does not have its existence managed by the object (class)
\item The part (member) does not know about the existence of the object (class)
\end{DoxyItemize}

Therefore\+:


\begin{DoxyItemize}
\item Typically use pointer or reference members that point to or reference objects that live outside the scope of the aggregate class
\item Not responsible for creating/destroying parts
\end{DoxyItemize}\hypertarget{class_sample_class_autotoc_md168}{}\doxyparagraph{Association}\label{class_sample_class_autotoc_md168}

\begin{DoxyItemize}
\item To qualify as an association, an object and another object must have the following relationship\+:
\item The associated object (member) is otherwise unrelated to the object (class)
\item The associated object (member) can belong to more than one object (class) at a time
\item The associated object (member) does not have its existence managed by the object (class)$\ast$
\item The associated object (member) may or may not know about the existence of the object (class)
